The problem worth solving

Health data has a trust problem

🏋 Logs vanish

Gyms have dead zones. Most trackers drop entries or duplicate them on flaky networks. Users stop trusting the app, then stop logging.

🔍 Search fights you

"chiken brest" returns nothing; your daily oatmeal is 6 taps deep. Friction kills the logging habit that drives every outcome.

🧮 Numbers wobble

Same meal, different totals across screens — rounding scattered through UI code. Correctness must live in one tested engine.

Architecture

Offline-first, one source of truth per concern

UI (Next.js)→ IndexedDB
entries + outbox→ Sync API
idempotency keys→ PostgreSQL
revisions, FTS ⇄ARQ worker
jobs, retries

Writes are queued mutations

Every diary write lands locally first (instant UI), then drains through POST /sync/push with a stable idempotency key. Retries, double-submits, and replay after crash are all no-ops by construction.

Conflicts are surfaced, never merged silently

Per-entry revisions detect concurrent edits from other devices. The server returns its copy; the user picks a winner in a side-by-side dialog.

Deep dive 1 · Sync engine

The same request twice = one record. Proven.

Idempotency gate (Postgres)

INSERT … ON CONFLICT DO NOTHING on the key table decides atomically whether a mutation is new. Replays return the stored result as duplicate. Same client entry-id with a lost key? Still one row.

Optimistic concurrency

Updates carry baseRevision. Stale writes get conflict + the server copy. Deletes are tombstones so every device converges. All paths covered by 19 API tests.

Deep dive 3 · Correctness & jobs

One nutrition engine, two languages, same vectors

Deterministic math

TS and Python implementations run identical test vectors (2 eggs = 155 kcal; meal-prep recipe = 536 kcal/serving; half-up rounding incl. the 2.675 float trap). Rounding happens only at persistence boundaries.

Observable background jobs

ARQ on Redis: exports, account purge, weekly reports, popularity refresh — with exponential-backoff retries and a job_runs history table where the final failure is the dead-letter record.

AI, used responsibly

"2 eggs and toast" → structured, confirmable entries

Boundaries

  • LLM (Groq · Llama 3.3 70B) extracts structure only — items resolve through the same search as manual queries
  • User confirms every item; AI never writes to the diary
  • Timeout/outage → graceful degraded fallback to manual search
  • Per-user rate limits + privacy toggle; no health claims

Verified live

"a slice of whole wheat toast" → the model normalizes toast → bread, matched with HIGH confidence — better recall than the deterministic parser, with the same safety rails. App remains fully functional with AI disabled.
